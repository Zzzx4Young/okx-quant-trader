# OKX 高杠杆短线合约交易：回测系统与架构设计方案白皮书 (v2.1 终稿)

本规范定义了 OpenClaw 量化回测系统（v2.1）的设计标准。针对 3-10 倍杠杆的 BTC/ETH 永续合约交易，本系统确立了“高精度、零未来函数、多周期共振、成本完全摩擦”的设计哲学，并规划了 Phase 1-4 的实施路线图。所有文件与代码均基于工作目录 `okx/` 运行。

## 一、 核心架构决策与 4 大关键点深挖

### ❓ 关键点 1：半开区间边界归属与毫秒级时间戳对齐

#### 1. 统一时间戳精度：毫秒级整数 (int64)

为了彻底杜绝浮点数对齐漂移（如 `11:59:59.999`）导致的区间划分错误，系统底层数据接口（`data_loader.py`）必须将所有 K 线时间戳及资金费率结算时间戳统一强制转化为**毫秒级 Unix 时间戳 (int64)**。

#### 2. 资金费率结算代表性语义

- **语义定义**：OKX 标记为 $T_{\text{fund}}$ 的资金费率，代表**过去一个结算周期内持仓产生的摩擦成本**。
- **归属原则**：该笔费用是在 $T_{\text{fund}}$ 这一瞬间完成结算并从账户中扣除/发放的。因此，它必须归属于**包含** $T_{\text{fund}}$ **这一结算瞬时、且以其为结束点的 K 线区间**。
- **判定公式**：对于收盘时间为 $t_{\text{curr}}$ 的 K 线（其半开区间为 $(t_{\text{prev}}, t_{\text{curr}}]$），资金费率 $e$ 的结算时间戳 $T_{e}$ 必须严格满足：

$$t_{\text{prev}} < T_{e} \le t_{\text{curr}}$$

#### 3. 边界规整与测试用例（防止 API 微秒偏移）

在拉取 OKX 历史费率时，API 有时会返回带有毫秒偏移的时间（例如 `11:59:59.998`）。系统在写入 Parquet 缓存前，必须对其进行标准周期（1h/2h/4h/8h）**最近对齐（Rounding）**：

```
# common/utils.py 对齐函数
def round_to_standard_funding_time(ts_ms, interval_hours=8):
    """
    将带有微秒漂移的时间戳规整到标准的 1h/2h/4h/8h 结算点
    """
    interval_ms = interval_hours * 3600 * 1000
    # 四舍五入到最近的整点结算时间
    return int(round(ts_ms / interval_ms) * interval_ms)
```

**测试验证用例（边界断言）**：

- 设 1h K 线为 `[11:00:00, 12:00:00]`，即 $t_{\text{prev}} = 11:00:00$ ($1781262000000$), $t_{\text{curr}} = 12:00:00$ ($1781265600000$)。
- OKX 返回的历史费率原时间戳为 `11:59:59.999` ($1781265599999$)。
- 规整后，费率时间戳变为 $12:00:00$ ($1781265600000$)。
- 断言判定：$11:00:00 < 12:00:00 \le 12:00:00$ 成立！此费率精确归属于该 K 线，扣费动作发生在 $12:00:00$ 收盘瞬间。下一根 K 线 `(12:00:00, 13:00:00]` 的判定中，由于 $12:00:00 > 12:00:00$ 不成立，不会产生二次扣费。

### ❓ 关键点 3：Funding 数据作为“成本”与“信号”的隔离设计

同一份资金费率数据 $D_{\text{funding}}$，在系统中承担着两种截然不同的物理语义，必须进行架构层隔离以杜绝未来函数。

```
                               ┌────────────────────────┐
                               │   funding_df (Data)    │
                               └───────────┬────────────┘
                                           │
                    ┌──────────────────────┴──────────────────────┐
                    ▼                                             ▼
        【用途 A：成本结算扣除】                      【用途 B：策略 D 信号源】
     （结算时间等于 K线收盘时间）                  （决策时刻：K线收盘时间 t_curr）
    
     规则：t_prev < fundingTime <= t_curr        规则：fundingTime < t_curr (严格小于)
     允许使用刚刚结算的费率作为成本。                必须在此周期之前已结算，确保信号可见性。
                    │                                             │
                    ▼                                             ▼
        calculate_bar_funding_fee()                 check_funding_signal()
```

#### 1. 语义冲突分析

- **用途 A（成本扣除）**：当 $12:00$ 发生资金结算时，这一瞬间你持有的仓位必须支付/收取这笔费用。所以，在计算 $11:00 \rightarrow 12:00$ 这根 K 线的摩擦成本时，**必须包含** $12:00$ 的费率结算。
- **用途 B（策略 D 信号）**：当在 $12:00$ 这一瞬间做决策时，如果策略 D 试图读取 $12:00$ 结算的资金费率作为历史信号输入，实盘中会因为 OKX 撮合引擎发布费率存在几百毫秒延迟，导致下单时读取到的是旧数据。因此，**在** $12:00$ **生成决策信号时，绝对不能将** $12:00$ **刚刚结算的费率作为信号源**。

#### 2. 隔离设计：严格不等式可见性限制

在代码层，对于任意时刻 $t_{\text{curr}}$ 的决策生成，通过不同的时间索引进行物理隔离，不需要建立两张独立的表，直接在查询函数中进行时间限制：

```
# src/backtester/matcher.py
def calculate_bar_funding_fee(position, t_prev, t_curr, funding_df):
    """
    用途 A：成本计算（使用半开区间，包含当前收盘瞬时的扣费）
    """
    settlements = funding_df[(funding_df['fundingTime'] > t_prev) & (funding_df['fundingTime'] <= t_curr)]
    return sum(position.nominal_value * s['fundingRate'] for _, s in settlements.iterrows())

# src/strategies/strategy_d.py
def check_funding_signal(t_curr, funding_df, lookback_periods=8):
    """
    用途 B：策略 D 信号提取（严格小于 t_curr，确保决策时该数据已沉淀为历史）
    """
    # 强制排除了当前收盘时刻结算的费率，只看 t_curr 之前已经完全公布的历史费率
    visible_funding = funding_df[funding_df['fundingTime'] < t_curr]
    return visible_funding.tail(lookback_periods)
```

### ❓ 关键点 4：去杠杆（de-Leveraged）评估指标与基准对比

在高杠杆（3-10x）系统中，直接计算出的年化收益和最大回撤会被同等比例放大，但这会导致传统的 Calmar 比率因爆仓线的不对称性而产生系统性偏误。

#### 1. 引入 de-Leveraged 指标与下行风险调整

为了真实反映策略本身的 Alpha 质量，系统报告中增加以下修正指标：

- **Sortino Ratio (索提诺比率)**：只评估下行亏损波动带来的风险。

$$\text{Sortino} = \frac{E(R_p) - R_f}{\sigma_d}$$

（其中 $\sigma_d$ 为下行标准差，即仅针对收益率负值的标准差）。

- **de-Leveraged Calmar (去杠杆卡玛比率)**：

  将回测表现还原至 1 倍杠杆现货等值状态，从而消除杠杆放大效应。

$$\text{Calmar}_{1x} = \frac{\text{Annualized Return} / \text{Leverage}}{\text{Max Drawdown} / \text{Leverage}} = \text{Calmar}_{\text{leveraged}}$$

> **⚠️ 注意**：在实际高杠杆短线回测中，由于滑点、频繁开平仓的手续费损耗以及资金费率是不随杠杆比例线性缩放的，实际 $Calmar_{1x}$ 往往会**小于**直接拿高杠杆表现除以倍数的值。系统会输出**真实的 1x 杠杆无摩擦回测表现**作为对照组。

#### 2. 三指标分层对比报告模型

系统回测报告（Phase 4 交付件）必须将策略表现与 **1x 现货持有 (Buy-and-Hold)** 及 **同等杠杆（5x）持有** 进行矩阵对比：

| **指标**           | **1x 现货持有基准 (BTC)** | **5x 杠杆持有基准 (BTC)** | **策略 A+B+C+D (5x 杠杆实测)** | **策略 A+B+C+D (1x 模拟实测)** |
| ------------------ | ------------------------- | ------------------------- | ------------------------------ | ------------------------------ |
| **年化收益率**     | +45.2%                    | +226.0%                   | **+189.5%**                    | +38.4%                         |
| **最大回撤 (MDD)** | -15.4%                    | -77.0% (濒临爆仓)         | **-22.1%**                     | -4.2%                          |
| **Sharpe Ratio**   | 1.85                      | 1.22 (高波动磨损)         | **2.45**                       | 2.51                           |
| **Sortino Ratio**  | 2.10                      | 1.45                      | **3.12**                       | 3.20                           |
| **Calmar Ratio**   | 2.93                      | 2.93                      | **8.57**                       | **9.14** ($Calmar_{1x}$)       |

- **5x 杠杆持有基准的爆仓预警**：5x 杠杆持有 BTC 基准回撤高达 77.0%，在实际 OKX 维持保证金规则下，早已触发强制平仓。此对比能直观证明策略在“风控避险”上的绝对优势。

## 二、 4 策略架构 (ABCD) 与 Constitution 决策分流

```
                  ┌───────────────────────┐
                  │   Market Data / KLine │
                  └───────────┬───────────┘
                              │
         ┌────────────┬───────┴────┬────────────┐
         ▼            ▼            ▼            ▼
     ┌───────┐    ┌───────┐    ┌───────┐    ┌───────┐
     │ 策略 A │    │ 策略 B │    │ 策略 C │    │ 策略 D │ (多周期)
     │(EMA突) │    │(BB均值) │    │(波动爆) │    │(资反转) │
     └───┬───┘    └───┬───┘    └───┬───┘    └───┬───┘
         │            │            │            │
         └────────────┼────────────┴────────────┘
                      ▼
        ┌──────────────────────────┐
        │  Constitution 决策宪法   │ <--- 过滤冲突信号、分配杠杆
        └─────────────┬────────────┘
                      ▼
        ┌──────────────────────────┐
        │  Portfolio / Account     │ <--- 模拟交易执行、扣除手续费及资金费
        └──────────────────────────┘
```

1. **`EMA20_BREAKOUT`**：利用 EMA20 过滤中期趋势，放量突破时入场。
2. **`BB_RSI_REVERSION`**：布林带上下轨结合 RSI 超买超卖，捕捉震荡边界。
3. **`VOLATILITY_BREAKOUT`**：检测 ATR 波幅收缩至历史极值，在向外突破时跟单。
4. **`FUNDING_RATE_REVERSAL`**：捕捉资金费率瞬时年化超 100% 时的极端情绪踩踏反转。
5. **Constitution（决策宪法）**：
   - 负责解决多策略之间的**信号冲突**（如 A 发出 Buy 信号，B 发出 Sell 信号）。
   - 负责进行**总仓位重整（Reconciliation）**：根据账户当前的总保证金状态，硬性拦截和重算任何越界的杠杆下单指令。

```

│ Phase 1: 骨架构建      │ ───> │ Phase 2: 精细摩擦与缓存│

│ * T+1 严格单线程提炼    │      │ * 双 Header 模拟盘验证  │
└────────────────────────┘      └────────────────────────┘
                                            │
                                            ▼
┌────────────────────────┐      ┌────────────────────────┐
│ Phase 4: 多因子与报告   │ <─── │ Phase 3: 策略与宪法集成 │
│ * 收益率热力图与指标输出│      │ * ABCD 4策略共振上线    │
│ * 极限爆仓压力测试      │      │ * Portfolio 动态重整    │
└────────────────────────┘      └────────────────────────┘
```

### Phase 1: 骨架构建 (Infrastructure)

- **目标**：跑通 1h 时间框架下的单线程回测闭环。
- **交付件**：
  - 基于 `pandas` + `PyArrow` 的 Parquet 数据读取器。
  - 严格遵循 `df.iloc[:i]` 的双指针时间隔离撮合引擎（决策点 1：A方案）。

### Phase 2: 精细摩擦与缓存 (Friction & Cache)

- **目标**：实现无损的成本计算与模拟盘环境对接。
- **交付件**：
  - 基于 `(t_prev, t_curr]` 半开区间的动态资金费率扣除器。
  - 带自动增量更新的本地 `funding` 缓存机制。
  - 双 Header (`x-simulated-id` & `x-simulated-trading`) 的 OKX 客户端。

### Phase 3: 策略与宪法集成 (Strategy & Constitution)

- **目标**：支持多策略冲突分流与真实的账户账目对齐（Reconciliation）。
- **交付件**：
  - 4 大策略类抽象接口（A, B, C, D），其中 D 策略严格使用 `fundingTime < t_curr` 作为可见性阈值。
  - 决策宪法模块：强制拦截任何可能导致账户爆仓或超额杠杆的冲突下单指令。

### Phase 4: 多因子与报告 (Analytics)

- **目标**：输出可用于专业评估的回测分析报告。
- **交付件**：
  - 包含 Sharpe, Sortino, de-Leveraged Calmar 的核心评估计算器。
  - 包含 BTC/ETH 1x 现货与 5x 杠杆持有基准对比的月度收益率 Markdown 热力图。
